- **カテゴリ:** セキュリティ / 暗号技術
- **プロジェクト:** 情報処理安全確保支援士

## [002] DH法（Diffie-Hellman鍵交換）
- **日付:** 2026-02-16(月)
- **カテゴリ:** セキュリティ / 暗号技術
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:** DH法は、公開しても良い素数と整数を作って、公開しない乱数をそれぞれで作り素数と整数から共通鍵を作り出すハイブリッド暗号方式
- **正誤:** 一部修正
- **補足:**
  - DH法は「鍵交換プロトコル」であり、ハイブリッド暗号方式ではない
  - 用語の整理：
    - 鍵交換: 共通鍵を安全に共有する方法（DH法、ECDH）
    - ハイブリッド暗号: 公開鍵で共通鍵を送り、その共通鍵でデータを暗号化（SSL/TLS）
  - DH法の仕組み：
    - 公開パラメータ（素数p、生成元g）
    - 秘密の乱数（a, b）を各自が生成
    - 公開値（g^a mod p, g^b mod p）を交換
    - 共通鍵（g^(ab) mod p）を各自が計算
  - 盗聴者は公開情報だけでは共通鍵を計算できない（離散対数問題）
  - DH法とハイブリッド暗号の共通点：どちらも「共通鍵の配送問題」を解決する技術
- **実務での使用頻度:** 高（HTTPS通信のTLSでECDHが使われている）
- **関連:** [001]（証明書と組み合わせて使用）

## [003] ハッシュ関数の3つの性質
- **日付:** 2026-02-18(水)
- **カテゴリ:** セキュリティ / 暗号技術
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - 衝突発見困難性：メッセージからハッシュ関数を作った時にそのハッシュ関数と同じハッシュ関数を作るためにメッセージを探すことが困難
  - 第二現像計算困難性：メッセージからハッシュ関数を作った時に、別でハッシュ関数を探し続けて同じメッセージを見つけることが難しい
  - 原像計算困難性：ハッシュ関数から元のメッセージを算出することが難しい
- **正誤:** 一部修正
- **補足:**
  - 用語の修正：
    - 「ハッシュ関数を作る」→「ハッシュ値を計算する」
    - 「第二現像」→「第二原像」
  - 正しい理解：
    1. **原像計算困難性（一方向性）**
       - ハッシュ値 → メッセージを求めるのが困難
       - 実務: パスワード保存
    2. **第二原像計算困難性（弱衝突耐性）**
       - メッセージM1とそのハッシュ値H1が与えられた時、H1 = hash(M2)となる別のM2を見つけるのが困難
       - 実務: デジタル署名
    3. **衝突発見困難性（強衝突耐性）**
       - hash(M1) = hash(M2)となる任意のM1, M2のペアを見つけるのが困難
       - 実務: デジタル証明書
  - 難易度: 原像 > 第二原像 > 衝突（衝突は誕生日攻撃で意外と簡単）
  - アルゴリズムの現状：
    - SHA-1: 非推奨（衝突発見困難性が破られた）
    - SHA-2（SHA-256, SHA-384, SHA-512）: 現在の主流
    - SHA-3: 新世代だが性能が良いわけではない（SHA-2の代替案）
    - MD5: 非推奨だが一部で使用
- **実務での使用頻度:** 高（パスワードハッシュ、JWT署名、証明書検証で毎日使用）
- **関連:** [001]（証明書検証でSHA-256が使われる）, [002]（HMAC-SHA256がJWT署名に使われる）

## [006] S/Key（チャレンジレスポンス認証）
- **日付:** 2026-02-24(火)
- **カテゴリ:** セキュリティ / 認証
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  1. クライアントがマスターパスワードとシード値をN回ハッシュ化したものと、シード値をサーバーに登録する
  2. クライアントがサーバーに接続要求を出す
  3. サーバーは登録時に貰ったシード値とシーケンス番号をチャレンジとしてクライアントに送付する
  4. クライアントはマスターパスワードと貰ったシード値をシーケンス回数分ハッシュ化してワンタイムパスワードを生成する
  5. ハッシュ値をレスポンスとしてサーバーに返す
  6. サーバーは受け取ったOTPを1回ハッシュ化して登録済みのハッシュ値と比較し認証する
  7. シーケンスが0になったら新しいシードで最初から再登録する
- **正誤:** 正解（議論を経て理解）
- **補足:**
  - ワンタイムである理由：シーケンス番号が毎回1ずつ減るため、同じOTPは2度使えない
  - S/Keyで防げる攻撃：リプレイ攻撃（過去のOTPの再利用）
  - S/Keyで防げない攻撃：中間者攻撃（リアルタイムの横取り）→ TLSの役割
  - 再登録時は必ず新しいシードで登録する（シードを使い回すと過去の盗聴OTPが再利用される危険がある）
- **実務での使用頻度:** 低（現在はTOTP等に置き換わっているが、試験頻出）
- **関連:** [005]（ベーシック認証との対比）, [003]（ハッシュの一方向性が核心）

## [007] 生体認証（FRR/FARのトレードオフ）
- **日付:** 2026-02-25(水)
- **カテゴリ:** セキュリティ / 認証
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - 生体認証には指紋認証、静脈認証、虹彩認証、顔認証がある

- **正誤:** 一部修正
- **補足:**
  - FRR=FARとなる点を **EER（等エラー率）** といい、システムの精度比較に使用
  - 最大の弱点：生体情報はパスワードと違い漏洩後に変更不可能
- **実務での使用頻度:** 高（スマートフォン・入退室管理・本人確認で広く使用）
- **関連:** [005]（認証方式として）, [006]（OTPと組み合わせた多要素認証）

## [009] FIDO認証（パスワードレス認証）
- **日付:** 2026-02-25(水)
- **カテゴリ:** セキュリティ / 認証
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  1. クライアントは事前に公開鍵を認証サーバーに登録する
  2. オンラインサービスに認証要求を送ると、認証サーバーがチャレンジをクライアントに送る
  3. クライアントは生体認証/PINで秘密鍵を使える状態にし、チャレンジに署名する
  4. 署名済みチャレンジをレスポンスとして認証サーバーに返す
  5. 認証サーバーが公開鍵で検証し、成功すればサービスが使えるようになる
  - 生体情報/PINはデバイス外に出ない（サーバーに送らない）
- **正誤:** 一部修正
- **補足:**
  - 「ロック」ではなく「**署名**」が正確（ロック=暗号化、署名=本人証明）
  - 生体情報をサーバーに送らない理由：サーバー漏洩リスクを排除するため。生体情報は変更不可能なので漏洩すると永久に使えなくなる
  - **弱点①：** デバイス紛失・故障で秘密鍵が失われ、認証不能になる
  - **弱点②：** 初回登録時はパスワード等の別の認証に依存する（そこが弱いと攻撃者が自分の公開鍵を登録できる）
  - 弱点②の対策例：TOTPや多要素認証で初回登録を強化する（[006]との組み合わせ）
- **実務での使用頻度:** 高（パスキー（Passkey）として主要ブラウザ・OS・サービスに普及中）
- **関連:** [006]（TOTPで初回登録を強化）, [007]（生体認証がローカル認証手段として使われる）

## [011] SSO - ケルベロス認証
- **日付:** 2026-02-26(木)
- **カテゴリ:** セキュリティ / 認証 / SSO
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - KDC = AS（認証・TGT発行）+ TGS（ST発行）の総称。レルムを一元管理
  - 管理者がKDCにパスワード由来の長期鍵を事前登録（パスワード自体はネットワークを流れない）
  - 認証時：クライアント → AS（ユーザー名のみ送信）→ TGT取得 → TGSからST取得 → サービスアクセス
  - SSOの実現：TGTがクライアントにキャッシュされるため、2回目以降はTGT→ST→サービスアクセスで認証不要
  - パスワード変更：旧パスワードの長期鍵で暗号化して新パスワードを送信
- **正誤:** 一部修正
- **補足:**
  - 「KDCにアクセス→ASに引き渡す」は不正確。クライアントはKDC内のASに直接通信する
  - 「KDCがログイン情報を覚えている」は誤り。SSOはTGTのクライアントキャッシュで実現
  - 弱点①：KDCが単一障害点（KDC停止でレルム全体が認証不能）
  - 弱点②：時刻同期必須（5分以上ずれると認証失敗）
- **実務での使用頻度:** 高（Active Directoryの認証基盤として企業で広く使用）
- **関連:** [009]（FIDOと同様、初回登録は別手段が必要）

## [012] SSO - SAML
- **日付:** 2026-02-26(木)
- **カテゴリ:** セキュリティ / 認証 / SSO
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  1. クライアント → SP（サービスプロバイダ）にアクセス要求
  2. SP → クライアント：HTTPリダイレクト（「IdPに行って」と誘導）
  3. クライアント → IdP（アイデンティティプロバイダ）に認証要求
  4. IdPが認証OK → アサーション（XML形式の認証済み証明書）をクライアントに送信
  5. クライアント → SPにアサーションを提示 → サービス利用
  - SSOの実現：IdPがブラウザとのセッションを保持しているため、2回目以降は再認証なしでアサーションを発行
- **正誤:** 正解
- **補足:**
  - HTTPリダイレクト：「このURLではなく、こちらに行って」とブラウザに指示する仕組み（302応答）
  - アサーション：「この人は認証済み」というIdP発行のXML形式パスポート
  - IdPはSP側の認証方法を問わない（パスワード・FIDO・MFA何でもOK）
  - 弱点：アサーション盗聴リスク → 対策はHTTPS＋有効期限設定＋IdPによる署名
- **実務での使用頻度:** 高（企業向けSaaSのSSO基盤として広く採用）
- **関連:** [010][011]（同じSSOをリバースプロキシ・ケルベロスで実現する別方式）

## [013] OAuth 2.0（認可フレームワーク）
- **日付:** 2026-02-26(木)
- **カテゴリ:** セキュリティ / 認証 / 認可
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  1. リソースオーナー（ユーザー）はアカウント情報をリソースサーバーに登録しておく
  2. OAuthクライアントが認可サーバーに認可要求を送る
  3. 認可サーバーはリソースオーナーのブラウザにリダイレクトする
  4. リソースオーナーが「許可する」ことで認可グラントの作成を認可する
  5. 認可サーバーが認可グラントを作成してOAuthクライアントに送る
  6. OAuthクライアントが認可グラントを認可サーバーに提出する
  7. 認可サーバーがアクセストークンをOAuthクライアントに発行する
  8. OAuthクライアントがアクセストークンでリソースサーバーにアクセスしデータを取得する
- **正誤:** 正解（議論を経て理解）
- **補足:**
  - OAuthは「認可」のプロトコル（認証ではない）
  - 4つの登場人物：
    - リソースオーナー：データの持ち主（ユーザー）
    - OAuthクライアント：データにアクセスしたいアプリ
    - 認可サーバー：アクセスを許可する機関
    - リソースサーバー：データを持っているAPI
  - 「Googleでログイン」はOAuthではなくOpenID Connect（OIDC）が正確。OIDCはOAuth 2.0を拡張してIDトークンを追加したもの
  - Googleの例：TwitterがGoogleの認可サーバーに認可要求 → ユーザーが許可 → TwitterがGoogleのAPIでユーザー情報を取得
- **実務での使用頻度:** 高（SNSログイン・API連携・外部サービス統合で広く使用）
- **関連:** [012]（SAMLと同様にサービス間の認証・認可連携）

- **日付:** 2026-02-27(金)

- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  1. 送信者がメッセージをMAC鍵でハッシュ化してMACを作る
  2. メッセージとMACを受信者に送る
  3. 受信者が受け取ったメッセージとMAC鍵でハッシュ値を作る
  4. 受け取ったMACと作ったMACを比較して完全性を検証する
  - MACは完全性を担保するもので、暗号化（機密性）は別の仕組み
- **正誤:** 一部修正
- **補足:**
  - 「秘密の情報」ではなく「**MAC鍵**」が正確な用語
  - HMACに暗号化は含まれない。HMAC = Hash(MAC鍵, メッセージ) のみ
  - 弱点①：MAC鍵が漏洩すると誰でも正当なMACを作れる
  - 弱点②：**否認防止ができない**（送信者と受信者が同じMAC鍵を持つため、誰が送ったか第三者に証明できない）
  - 暗号化とMACは役割が違うため、TLSでは両方を併用する
- **実務での使用頻度:** 高（TLS通信・APIリクエスト検証で広く使用）
- **関連:** [003]（ハッシュ関数がMACの基盤）, [001]（証明書検証と同じ完全性確認の考え方）

## [015] デジタル署名
- **日付:** 2026-02-27(金)
- **カテゴリ:** セキュリティ / 暗号技術
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  1. 送信者の公開鍵を事前に受信者へ送る
  2. 送信者はメッセージをハッシュ化して秘密鍵で暗号化することでデジタル署名を作る

  5. 平文のメッセージをハッシュ化して、復号したハッシュ値と比較することで完全性を担保する

## [017] サーバー証明書（DV/OV/EV・ワイルドカード・SAN）
- **日付:** 2026-02-28(土)

- **実務での使用頻度:** 高（HTTPS通信で毎回使用）

## [018] PKI・証明書検証・CRL・OCSP
- **日付:** 2026-03-02(月)
- **カテゴリ:** セキュリティ / 暗号技術
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - **PKI：** サーバー証明書はCAの署名付き。ルートCA→中間CA→サーバー証明書の階層構造。ルートCAは自己署名だが、ブラウザ・OSに事前組み込みされているため信頼できる
  - **証明書検証：** 上位CAの公開鍵で下位CAを検証する連鎖。実質ルートCAがすべてを担保
  - **CRL：** 有効期間内に失効した証明書のリスト。ダウンロード後に失効した場合は気づけない
  - **OCSP：** ブラウザ（クライアント）がOCSPレスポンダに直接問い合わせてリアルタイムで失効確認
  - **OCSPステープリング：** サーバーがOCSPレスポンダにアクセスしてCAの署名付きOCSP応答をキャッシュ。ブラウザとの通信時にサーバー証明書と一緒に送付。ブラウザはOCSPレスポンダに直接アクセス不要
- **正誤:** 一部修正
- **補足:**
  - CRLは「期限切れ」ではなく「**失効**（有効期間内の強制無効化）」のリスト
  - OCSPのsoft-fail（フェールソフト）問題：OCSPレスポンダがDoS攻撃等で停止すると失効確認をスキップして接続が許可される。フェールセーフにしない理由は可用性のトレードオフ（OCSPが落ちるたびに全HTTPS接続が止まると困るため）
  - PKI最大の弱点：CAが侵害されると偽の証明書が大量発行され、ブラウザは気づけない（DigiNotar事件）
  - OCSPステープリングの利点：プライバシー保護（CAに閲覧履歴が漏れない）・高速化・OCSPレスポンダ停止耐性
- **実務での使用頻度:** 高（HTTPS通信の基盤として常時使用）
- **関連:** [001]（サーバ証明書の検証プロセス）, [016][017]（デジタル証明書・サーバー証明書）

## [019] 時刻同期型OTP（TOTP）
- **日付:** 2026-03-08(日)
- **カテゴリ:** セキュリティ / 認証
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - クライアントとサーバーで時刻を同期し、クライアントが時刻からOTPを作成してサーバーに送る
  - サーバーは同じ計算でOTPを作成し、クライアントのOTPと照合して検証する
- **正誤:** 一部修正
- **補足:**
  - OTP生成に必要なのは「時刻」だけでなく「**シークレットキー + 時刻**」（初回登録時にQRコード等で共有）
  - `OTP = HMAC(シークレットキー, 現在時刻)` の計算式
  - 時刻は通常**30秒単位**で区切られる（30秒以内は同じOTPが有効）
  - ±1ステップ（30秒前後）は許容するサーバーが多い（時刻ズレ対策）
  - これは「改ざん確認」ではなく「**本人認証**」（シークレットキーを持つ人だけが正しいOTPを計算できる）
  - **弱点①：シークレットキー漏洩** → 永続的にOTPを偽造できる
  - **弱点②：フィッシング耐性がない** → 偽サイトでOTPを入力させ、攻撃者がリアルタイムで正規サイトに転送できる（構造上防げない）
  - **弱点③：リプレイ攻撃耐性がない** → 盗聴したOTPを30秒以内に使われると防げない（構造上防げない）
  - FIDOはフィッシング耐性があるため、TOTPとの対比で試験に出やすい
- **実務での使用頻度:** 高（多要素認証のワンタイムパスワードとして広く使用）
- **関連:** [006]（S/Key：カウンタ型OTP）, [009]（FIDO認証との対比：フィッシング耐性の有無）

## [021] 3ウェイハンドシェイク / SYN flood攻撃
- **日付:** 2026-03-12(木)

- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - 3ウェイハンドシェイク：クライアントがSYNを送る → サーバーがSYN/ACKを返す → クライアントがACKを返す → TCPコネクション確立

  - SYN flood攻撃は1台でも可能なのでDoS攻撃。複数台から行えばDDoSになる

## [027] IDS/IPS 検知方式（ミスユース型・アノーマリ型）とFP/FN
- **日付:** 2026-03-20(金)
- **カテゴリ:** セキュリティ / ネットワーク
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - ミスユース型：シグネチャと照合して既知の悪意ある通信を検知。すぐ使えるが未知の攻撃（ゼロデイ）は検知できない
  - アノーマリ型：正常な通信を学習させ、通常と異なる動きを検知。学習期間が必要だがゼロデイ攻撃にも対処できる
  - フォールスネガティブ（FN）：感染しているのに感染していないと判断（見逃し）
  - フォールスポジティブ（FP）：感染していないのに感染していると判断（誤検知）
  - FPが多すぎると本物の脅威がアラートの海に埋もれて見逃される
  - FPとFNはトレードオフの関係であり、適切な割合で管理することが重要
- **正誤:** 一部修正
- **補足:**
  - ミスユース型 → FNが多くなりやすい（シグネチャにない攻撃を素通しする）
  - アノーマリ型 → FPが多くなりやすい（正常な通信を異常と誤判定しやすい）
  - ゼロデイ攻撃への対処：
    - ミスユース型：❌ リストにない攻撃は素通り
    - アノーマリ型：✅ 正常から外れれば既知・未知問わず検知できる
- **実務での使用頻度:** 高（セキュリティ製品の設計・運用で必須知識）

## [028] プロキシサーバー（フォワード・リバース）
- **日付:** 2026-03-22(日)
- **カテゴリ:** セキュリティ / ネットワーク
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - **フォワードプロキシ：** クライアントとインターネットの通信を代理
    - 通信経路：クライアント→FW→プロキシ→FW→インターネット
    - キャッシュ機能：過去にアクセスしたWebサーバーのレスポンスを保存・再利用
    - クライアント隠蔽：インターネット側からクライアントのIPが見えない（プロキシのIPしか見えない）
    - フィルタリング機能：悪意あるWebサーバーへの接続を制御
    - 認証機能：認証されたユーザーのみ外部接続を許可
  - **リバースプロキシ：** インターネットからサーバーへの通信を代理
    - 通信経路：インターネット→FW→プロキシ→FW→サーバー
    - 負荷分散：複数のバックエンドサーバーへ振り分け
    - サーバー情報隠蔽：バックエンドサーバーのIPが外部から見えない
    - SSL終端（TLSオフロード）：HTTPS→HTTPに復号し、バックエンドの暗号化負荷を軽減
    - WAF機能：HTTPリクエストの中身を検査して攻撃を検知・遮断
- **正誤:** 一部修正
- **補足:**
  - 認証機能の正確な説明：「プロキシへの接続時にユーザー認証を求め、許可されたユーザーのみ外部接続を許可する」（感染拡大防止ではない）
  - **FWとプロキシの役割の違い（重要）：**
    - FW：ネットワーク層（L3/L4）で「IPアドレス・ポート」を見て誰と通信するかを制御
    - プロキシ/WAF：アプリケーション層（L7）でHTTPの中身を見て何を通信するかを制御
    - 層が違うため役割が異なり、どちらか一方では足りない → 多層防御として組み合わせる
  - FWにキャッシュ・WAFができない理由：FWはHTTPの中身を見ていないため
- **実務での使用頻度:** 高（企業ネットワーク・Webサービスのセキュリティ設計で必須）
- **関連:** [022][023]（FWとの対比・多層防御）, [024]（WAFとの関係）

## [029] VPN（インターネットVPN・IP-VPN・IPsec）
- **日付:** 2026-03-24(月)
- **カテゴリ:** セキュリティ / ネットワーク
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - VPNは機密性・完全性・認証を担保した仮想的な専用ネットワーク
    - 機密性：通信を暗号化
    - 完全性：改ざんの検証
    - 認証：通信相手の正当性確認
  - インターネットVPN：一般のインターネット回線を使用。暗号化・認証・改ざん検知で安全性を確保
  - IP-VPN：通信事業者の閉域網を使用。品質が担保されよりセキュア
  - IPsecのモード：
    - トンネルモード：元のIPヘッダ＋データを丸ごと暗号化し新しいIPヘッダを付与。拠点間VPN（VPN機器同士）で使用
    - トランスポートモード：データのみ暗号化・元のIPヘッダは残る。ホスト間直接通信やL2TPなど他のトンネリング技術と組み合わせる場合に使用
- **正誤:** 一部修正
- **補足:**
  - VPNの3要素に「認証」を追加（機密性・完全性だけでなく認証も含む）
  - L2TP/IPsecの場合：L2TPがトンネリングを担い、IPsecはトランスポートモードで暗号化を担当
  - トンネルモードの利点：元のIPヘッダも暗号化されるため宛先IPアドレスが外部に漏れない
  - トランスポートモードの注意：宛先IPアドレスは暗号化されないため盗聴される可能性がある
- **実務での使用頻度:** 高（拠点間接続・リモートアクセスで広く使用）
- **関連:** [022]（FWとの組み合わせでネットワーク設計）

## [030] IPsec（AH・ESP・IKE・SA）
- **日付:** 2026-03-24(月)
- **カテゴリ:** セキュリティ / ネットワーク
- **プロジェクト:** 情報処理安全確保支援士
- **自分の理解:**
  - IPsec（Security Architecture for Internet Protocol）：機密性・完全性・認証を担保する通信アーキテクチャ
  - AH（Authentication Header）：認証・改ざん検知のみ。暗号化なし
  - ESP（Encapsulating Security Payload）：認証・改ざん検知・暗号化あり
  - IKE（Internet Key Exchange）：鍵交換プロトコル。SAを確立するために使う
  - SA（Security Association）：暗号アルゴリズム・鍵などセキュリティパラメータの合意状態（プロトコルではない）
  - 暗号化できる環境 → ESP + IKE、暗号化規制がある地域 → AH + IKE
- **正誤:** 一部修正
- **補足:**
  - SAはプロトコルではなく「2者間のセキュリティ設定の合意状態」。IKEを使ってSAを確立する
  - SAの種類（参考）：
    - ISAKMP SA（フェーズ1）：制御チャネル確立。双方向1本。鍵交換・認証情報を通信
    - IPsec SA（フェーズ2）：実際のデータ暗号通信。片方向2本（送受信で別々）
  - **試験での優先度：** AH/ESPの違い・トンネル/トランスポートモードが必須。ISAKMP SAとIPsec SAの区別は出題頻度低め
- **実務での使用頻度:** 高（VPN実装の基盤プロトコル）
- **関連:** [029]（VPN・トンネルモード・トランスポートモード）
