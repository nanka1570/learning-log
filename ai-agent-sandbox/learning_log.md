### 001 CI/CD
- **日付:** 2026-02-23(月)
- **カテゴリ:** AI・エージェント / ツール
- **プロジェクト:** AI-Agent-Sandbox（実践ClaudeCode入門 読書）
- **自分の理解:** git pushするコマンドがCLIだと使えるから、AI AgentがCI/CDパイプラインの起点（push）まで担当でき、その後のテスト・ビルド・デプロイが全自動で動く
- **正誤:** 正解
- **補足:**
  - CI（Continuous Integration）: pushされたらサーバー側（GitHub Actions等）で自動テスト・ビルドを実行する仕組み
  - CD（Continuous Delivery/Deployment）: テスト通過後に自動デプロイする仕組み（例: Vercel）
  - CLI型エージェントはコマンドを直接実行できるため、git pushまで担当でき、CI/CDと組み合わせることで「指示だけで本番反映まで全自動」が実現できる
  - hooks（例: Stop hook）はCI/CDとは別物。ローカルでの作業リマインダーや自動チェックに使う
- **実務での使用頻度:** 高
- **関連:** なし

### 002 1stパーティ / 3rdパーティ
- **日付:** 2026-02-23(月)
- **カテゴリ:** AI・エージェント / ツール
- **プロジェクト:** AI-Agent-Sandbox（実践ClaudeCode入門 読書）
- **自分の理解:** 1stパーティは作った会社自身が提供しているもの。Claude CodeはAnthropicが作ったClaude modelをAnthropicが作ったCLIで使えるので1stパーティ。
- **正誤:** 正解
- **補足:**
  - 1stパーティ: 作った本人が提供（例: AnthropicがClaude Codeを提供）
  - 3rdパーティ: 第三者（外部）が提供（例: 他社がClaudeを組み込んだツールを提供）
  - 2ndパーティはソフトウェア業界ではほとんど使われない（ゲーム業界固有の概念）
  - 1stパーティの利点: 最新モデルへ直接アクセス、新機能対応が早い、最適化が深い
- **実務での使用頻度:** 低（概念理解として重要）
- **関連:** なし

### 005 アンビエントエージェント
- **日付:** 2026-02-24(火)
- **カテゴリ:** AI・エージェント / 設計パターン
- **プロジェクト:** AI-Agent-Sandbox（実践ClaudeCode入門 読書）
- **自分の理解:** イベントを起点にしてAIに実行させることで、毎回指示を出すより時間が短縮できる
- **正誤:** 一部修正
- **補足:**
  - 時間短縮は正しいが、本質は「人間の認知リソースの解放」
  - 従来：人間が気づいて指示 → AIが実行
  - アンビエント：イベントを自動検知 → AIが実行（人間の介入不要）
  - 追加の価値：複数イベントを並列処理できる／人間が離席中も動く／「あれをやらなきゃ」と覚えておく必要がなくなる
  - Claude Code Actionはアンビエントエージェントの具体的な実装例（GitHubイベント起点）
- **実務での使用頻度:** 高
- **関連:** 004（Claude Code Action）

### 006 AI Agentの動作原理（5要素ループ）
- **日付:** 2026-02-24(火)
- **カテゴリ:** AI・エージェント
- **プロジェクト:** AI-Agent-Sandbox（実践ClaudeCode入門 第6章）
- **自分の理解:** AI Agentは環境・知覚・記憶・推論・行動の5つで動いている。環境で行動することで、それを知覚し、記憶を確認したうえで推論し、ツールを選択して行動を決める。これをループし続ける仕組み。
- **正誤:** 正解
- **補足:**
  - 5要素の役割: 環境（エージェントが働く場所）、知覚（環境からの入力を受け取る）、記憶（過去の情報・コンテキスト）、推論（次の行動を決める思考）、行動（ツール呼び出し等の実行）
  - 「ツールを選択する」と「行動する」は両方とも「行動」フェーズの内訳。5要素の区分としては行動1つに含まれる
  - このループ構造を意識すると、エージェントの挙動（なぜ失敗するか、どう改善するか）が理解しやすくなる
- **実務での使用頻度:** 高（設計・デバッグの思考基盤）
- **関連:** 007（コーディングエージェントの3つの問題）

### 007 コーディングエージェントの3つの問題
- **日付:** 2026-02-24(火)
- **カテゴリ:** AI・エージェント
- **プロジェクト:** AI-Agent-Sandbox（実践ClaudeCode入門 第6章）
- **自分の理解:** ①過去の指示を忘れる（contextの限界＋contextロット）、②意図した動作をしない（隠れた期待の未明示・矛盾したドキュメント）、③外部システムと適切に連携できない（MCPがあっても適切なツール選択は保証されない）
- **正誤:** 正解
- **補足:**
  - ①Contextロット: contextが増えると圧縮され、初期の詳細な取り決めが失われる。後半になるほど品質が落ちる
  - ②「隠れた期待」とはユーザーが当然だと思って言語化していない前提条件のこと。明示しないとエージェントは独自解釈する
  - ②矛盾した指示: ドキュメントが散在して内容が食い違うと、エージェントはどれを優先するか判断できなくなる
  - ③MCPは「標準的なツール呼び出しプロトコル」を提供するが、「どのツールをいつ使うか」の判断はエージェントの推論に依存する
  - これら3問題への対策がスペック駆動開発・CLAUDE.md・適切なMCP設計につながる
- **実務での使用頻度:** 高（エージェント設計の落とし穴として常に意識）
- **関連:** 003（スペック駆動開発）、006（AI Agentの動作原理）

### 008 コーディングエージェントの3つの問題への解決策
- **日付:** 2026-03-01(日)
- **カテゴリ:** AI・エージェント / Claude Code
- **プロジェクト:** AI-Agent-Sandbox（実践ClaudeCode入門 読書）
- **自分の理解:**
  - ①忘れる問題: 作業ログファイルをCLAUDE.mdで指示して維持させる／フェーズ区切りで/compact・/clearを使う／サブエージェントでcontextを分配する
  - ②意図通りに動かない問題: スラッシュコマンドでワークフローの順番を明示／スキルで実行ルールを段階的に明示／Hooksでプログラムとして絶対禁止・絶対実行を保証する
  - ③MCPが意図通り使われない問題: Skillsで「いつ何を使うか」のルールを定義してMCPと組み合わせる／CLIで操作パターンを標準化する／CLIのセキュリティリスクはsandboxing・DevContainerで制御する
- **正誤:** 正解（一部補足あり）
- **補足:**
  - ①の「作業ログファイル」: 正式名ではなくCLAUDE.mdへの指示で任意名（status.md等）のファイルを維持させる手法
  - ①の/compact vs /clear: /compactは重要情報を保持しつつ圧縮、/clearは完全リセット。使い分けが重要
  - ②の核心: スラッシュコマンド（順番）、スキル（ルール）、Hooks（保証）という役割分担。Hooksは「プロンプトは確率的、Hooksはプログラム的」が本質
  - ③のCLI: デプロイ等の複雑な操作を1コマンドに封じ込め、Claudeが「このコマンドを叩けばよい」という形にする手法
- **実務での使用頻度:** 高
- **関連:** 007（コーディングエージェントの3つの問題）
